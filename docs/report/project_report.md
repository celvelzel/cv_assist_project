# CV Assist: 视觉辅助系统项目开发报告

## 1. 项目开发过程中的困难及解决方法

在CV Assist系统的开发过程中，我们遇到了几个关键的技术挑战。为了实现一个稳定、实时、对视障用户友好的系统，我们对这些问题进行了深入分析并提出了相应的解决方案。

### 1.1 音频处理与视频渲染循环的阻塞问题

**困难描述：**
在早期的系统版本中，语音合成（TTS）和视频流渲染（OpenCV `imshow`）在同一个主线程中执行。由于TTS播报（特别是使用本地 `pyttsx3` 或云端调用时）是一个耗时操作，这会导致在播放语音指导时，视频流完全冻结（Screen Freeze），严重影响了目标检测和手部追踪的实时性。

**解决方法：**
*   **解耦架构 (Decoupling)：** 我们重构了系统架构，将语音管线（Voice Pipeline）与主渲染循环（Render Loop）分离。
*   **异步音频处理：** 引入了异步或基于独立线程的TTS模块实现。通过将TTS任务放入后台队列处理，主线程可以无缝地继续处理摄像头帧并显示画面，从而保证了视频流的流畅和实时的空间引导。相关的修复体现在了 `feat: decouple voice pipeline from render loop` 提交中，并在后续针对Windows系统上的异步播放进行了专门的稳定性优化（`fix(tts): stabilize Windows async playback`）。

### 1.2 语音识别（ASR）的准确性与鲁棒性

**困难描述：**
视障用户在实际使用环境中，可能会遇到背景噪音、口音或者发音不够清晰等问题。单纯依赖 Whisper ASR 将语音转换为文本后，直接通过简单的正则匹配提取目标物体（如从“找爱手及”中提取目标），往往会导致错误的检测指令，系统难以找到正确的物体。

**解决方法：**
*   **引入 LLM 视觉解析器 (LLM Vision Parser)：** 我们创新性地引入了 **LLM 增强的语音识别功能**（✨ 新功能）。
*   **多模态上下文融合：** 当用户说出指令时，系统不仅获取 Whisper 转录的（可能包含错误的）文本，还会同时截取最近的 4 帧摄像头画面。将转录文本与视觉上下文一起发送给大语言模型（如基于 Poe API 的 deepseek-v3.2）。
*   **智能纠错：** LLM 根据当前用户视野中的物体，智能地纠正语音识别错误。例如，将“找爱手及”结合画面中的手机，正确推断出目标是“手机”（phone）。如果 API 调用失败或超时，系统设计了优雅的回退机制（Fallback），退回使用正则解析。这个改进极大地增强了指令解析的鲁棒性。

### 1.3 空间引导指令的“抖动”问题

**困难描述：**
当用户的手部在目标的边缘或某个对齐临界点附近微小移动时，简单的阈值判断会导致系统频繁地在两种指令间切换（例如，交替播报“向左移”和“向右移”），也就是发生指令振荡（Oscillation）。这会让视障用户感到非常困惑。

**解决方法：**
*   **滞回阈值算法 (Hysteresis-based guidance)：** 我们在 `GuidanceController` 中实现了基于滞回（Hysteresis）的引导逻辑。通过设置不同的触发阈值和释放阈值，为指令状态切换引入了一个“缓冲区”。这样可以有效过滤掉微小移动带来的抖动，保证语音指导指令的连贯性和稳定性。

### 1.4 多配置与跨平台部署的复杂性

**困难描述：**
系统整合了多个复杂的深度学习模型（OWL-ViT, MiDaS, MediaPipe, Whisper），不仅对硬件要求不一，而且不同使用场景（仅测试、仅TTS、完整语音交互等）需要的参数配置差异很大。此外，不同操作系统的部署也有细微差别。

**解决方法：**
*   **配置文件与预设 (Configuration Profiles)：** 重构了配置管理系统，外部化为 `config.yaml`。设计了级联加载优先级（环境变量 > profile overrides > 基础配置）。提供了如 `fast`, `balanced`, `no-voice` 等多种预设，方便用户一键切换场景。
*   **敏感信息隔离：** 使用 `.env` 文件隔离如 Poe API Key 和 MiMo TTS API Key 等敏感信息。
*   **一键部署脚本：** 编写了兼容 Windows 的 `run.bat` 和兼容 Linux/macOS 的 `run.sh`，自动处理环境创建、依赖安装和 UTF-8 编码设置等繁琐步骤，降低了用户的使用门槛。

## 2. 项目的亮点

### 2.1 创新的多模态辅助交互闭环
项目并非简单地堆砌计算机视觉模型，而是精心设计了一个完整的交互闭环：
1. **多模态输入**：用户通过语音（自然语言）输入意图，摄像头提供环境的视觉上下文。
2. **零样本开放词汇检测**：使用 OWL-ViT 模型，**无需为了新物体重新训练模型**，用户想找什么就说找什么（"a cup", "my keys"）。
3. **空间感知**：结合 MiDaS 单目深度估计获取距离信息。
4. **意图执行与反馈**：结合 MediaPipe 手部追踪，系统理解用户手部位置，并通过智能引导算法将纠正指令通过 TTS 语音实时反馈给用户。

### 2.2 LLM 视觉融合纠错 (LLM Vision Parser)
这是项目中最具创新性的亮点之一。传统的语音助手通常只是“语音到文本 -> 意图抽取”。本项目引入了**视觉上下文**来辅助纠正**语音识别错误**。让 LLM 看到用户看到了什么，从而更准确地猜出用户的真实意图，显著提升了在嘈杂环境或非标准发音情况下的用户体验。

### 2.3 极致的无障碍用户体验设计
*   **非阻塞的实时反馈**：系统保证了视频处理和音频播报的并行执行，指导声音不会带来画面和感知上的延迟。
*   **防抖动的引导逻辑**：引入工程上常用的滞回算法解决交互边界的指令跳动，指令清晰明确。
*   **高质量云端 TTS 支持**：除了本地基础 TTS，还集成了小米 MiMo 等高质量云端语音合成，提供更自然、舒适的听觉体验。

### 2.4 工程化与可维护性
*   **模块化架构**：检测器（Detector）、音频处理（Audio）、系统调度（System/Guidance）高度解耦。例如，新增一个 TTS 后端（如 `mimo_backend.py`）无需修改核心控制逻辑。
*   **可观测性**：引入了统一的日志记录和完善的帧率（FPS）监控（包括处理 FPS 和端到端 E2E FPS），便于性能调优和故障排查。
*   **完备的测试用例**：包含了冒烟测试、音频子系统测试、引导逻辑（滞回）测试、LLM视觉测试等，保证了代码重构和功能迭代的可靠性。

## 3. 额外加分项 (Bonus Points)

除了满足核心的项目要求，我们还额外探索并实现了以下高级特性，提升了系统的整体价值：

### 3.1 OWL-ViT v2 的支持
在原有 OWL-ViT 的基础上，增加了对更新版本 OWL-ViT v2 的支持（详见提交 `feat: add OWL-ViT v2 support`）。v2 版本通常在小目标检测和特征提取上具有更好的表现，这直接提升了系统在复杂环境下的目标定位能力。

### 3.2 灵活的 GPU 加速兼容
系统设计时充分考虑了硬件差异，能够智能探测并利用 GPU 环境（`device: auto` 配置）。针对实际环境中的 GPU 加速进行了优化，确保在使用较大模型（如 Whisper ASR 和 OWL-ViT）时，依然能够保持较高的处理帧率。

### 3.3 手势识别的融合
在 MediaPipe 的基础追踪之上，增加了对简单手势（张开、握拳、指向）的识别。虽然目前主要用于定位引导，但这一设计为未来扩展更丰富的交互语义（例如：握拳表示“抓取确认”，张开表示“取消搜索”）奠定了基础。

### 3.4 动态的帧跳跃优化 (Frame Skipping Optimization)
为了在性能较弱的设备上也能流畅运行，系统在配置文件中引入了 `skip_frames_detection` 和 `skip_frames_depth` 策略。由于手部移动是一个连续的高频动作，而环境和目标的相对位置变化相对低频，通过分别控制目标检测和深度估计的刷新频率，在不损失用户体验的前提下大幅降低了计算负载。

### 3.5 任务监控设计规划
在项目文档中包含了 `task monitoring design plans`（任务监控设计规划），这展示了团队不仅关注代码实现，也注重系统在运行时的状态监控和评估指标设计，体现了成熟的软件工程思维。
