# `[task]` 终端单行摘要字段说明

本文档说明 `core.system` 周期性输出的 **`[task]`** 日志行中各段含义。该行由 `utils/task_metrics.py` 中 `TaskMetricsCollector.build_terminal_summary()` 生成，字段之间用 ` | ` 分隔。

## 行结构概览

```
[timestamp] [core.system] [INFO] [task] | task_id | target | state(中文) | 时长与指标…
```

前缀 `[task]` 之后依次为：**任务标识** → **目标描述** → **任务状态** → **自任务激活起的运行时长** → **各类性能与资源指标**。

---

## 固定前置字段（无 `key=` 前缀）

| 位置 | 示例 | 含义 |
|------|------|------|
|标签 | `[task]` | 标识本行为任务监测摘要 |
| `task_id` | `task_0001` | 当前任务 ID（会话内递增） |
| `target_query` | `a phone` | 目标主体（英文/配置中的查询串） |
| `task_state` | `ready(就绪)` | 任务状态英文名 + 中文括号说明。常见值：`running` 进行中、`ready` 就绪、`idle` 空闲、`finishing` 结束中、`finished` 已结束（任务结束后本行不再输出） |

---

## 时长类 `key=value` 字段

### `21.2s`（紧跟在状态后的第一段带单位时长）

- **含义**：自 **`start_time`（任务正式激活）** 到 **当前摘要时刻** 的墙钟时间，**单位秒**，保留 **1 位小数**。
- **注意**：不是「从语音开始找目标」起的总时长；「开始寻找到激活」见下方 `s2act`。

### `s2act`

- **含义**：**search to activate**——从系统记录「开始寻找目标」起点到 **任务正式激活** 的耗时（秒）。
- **格式**：`s2act=4.2s`；若无法统计（例如立即激活路径）为 **`s2act=N/A`**。
- **精度**：1 位小数。

### `poe`

- **含义**：**Poe** 视觉解析链路耗时（若本任务语音流程中调用过 LLM/Poe 解析）。
- **格式**：未调用为 **`N/A`**；已调用为 `Xm.mms` 或 `X.Xs`（见下文「耗时显示规则」）。

### 最近一帧各阶段耗时（`det1` … `e2e1`）

均来自 **最新一帧** 的 `FrameMetrics`，反映**当前这一帧**各阶段耗时，非全任务平均。

| 字段 | 含义 |
|------|------|
| `det1` | 目标检测耗时 |
| `hand1` | 手部检测/跟踪耗时 |
| `d1` | 深度估计耗时 |
| `g1` | 引导计算耗时 |
| `proc1` | 单帧算法总处理耗时（与主循环统计中的「处理」口径一致） |
| `e2e1` | 端到端单帧循环耗时（含采集、处理、绘制、显示等） |

### `sup1`

- **含义**：语音确认后的 **抑制窗结束后的首帧**的 `process_time_ms`（若已记录）。
- **格式**：有值为 `Xm.mms` / `X.Xs`；未触发为 **`sup1=N/A`**。

### `tts1`（可选）

- **含义**：自任务激活起，到 **首次播报引导类指令（TTS）** 的时间偏移。
- **格式**：仅在已记录首次引导 TTS 时出现，例如 **`tts1=+13.7s`**（`+` 表示相对任务 `start_time` 的秒数，1 位小数）。

---

## 执行率与滑动窗口统计

### `det_run` / `depth_run`

- **含义**：截至当前，**实际执行** 目标检测 / 深度估计的帧数占 **总帧数** 的比例。
- **格式**：百分比字符串，**1 位小数**（如 `33.6%`），与跳帧策略有关，不一定为 100%。

### `fps_avg` / `e2e_avg`

- **含义**：最近帧统计中的 **平均处理 FPS** / **平均端到端 FPS**（与 `FrameMetrics` 中 `proc_fps_avg`、`e2e_fps_avg` 一致）。
- **格式**：数值，**1 位小数**；单位为 **帧/秒**，不是毫秒。

### `p50` / `p95` / `jitter`

- **数据来源**：任务运行期间 **处理耗时**（`process_time_ms`）的环形缓冲（最近若干帧）。
- **含义**：
  - `p50`：处理耗时 **中位数**；
  - `p95`：处理耗时 **95 分位**；
  - `jitter`：处理耗时的 **标准差**（波动程度）。
- **单位**：与 `_format_ms_display` 相同（见下节）。

### `cam_fail`

- **含义**：任务期间 **相机读帧失败** 累计次数。
- **格式**：**1 位小数**（如 `0.0`），与日志中其它数值风格统一。

---

## 资源采样（可选出现）

仅在成功采样到数据时追加：

| 字段 | 含义 |
|------|------|
| `cpu` | 进程 CPU 占用（%），1 位小数 |
| `mem` | 进程 RSS（MB），1 位小数 |
| `gpu` | `torch.cuda.memory_allocated()` 估算显存（MB），1 位小数 |
| `net↑` / `net↓` | 全机网络发送/接收吞吐（KB/s，相对上一采样间隔），各 1 位小数 |

---

## 耗时显示规则（`ms` 与 `s`）

对 `poe`、各 `*1` 阶段、`sup1`、`p50`、`p95`、`jitter` 等由 **`_format_ms_display`** 格式化的项：

- 内部以 **毫秒** 计量；
- **小于 1000 ms**：显示为 **`XXX.Xms`**（1 位小数）；
- **≥ 1000 ms**：显示为 **`X.Xs`**（换算为秒，1 位小数）。

`poe` 在值为 0 时固定为 **`0.0ms`**（不改为秒）。

---

## 相关配置

- 摘要输出周期由配置项 **`logging.task_metrics_interval_sec`** 控制（见 `config.py` /你的环境配置）。
- 若 **`logging.enable_task_metrics`** 为 false，则不会输出 `[task]` 行。

---

## 与 JSON 任务报告的区别

- **终端 `[task]` 行**：面向实时运维，**最近一帧** + **短窗口**统计为主，单位按上表规则在 `ms`/`s` 间切换。
- **`logs/task_metrics/*.json`**：任务结束后的结构化报告，耗时等多数字段以 **秒** 为主键名（如 `*_sec`），精度与终端行不完全相同；时间类字段的 `value` 多为 **HKT 可读字符串**。

如需扩展字段，请以 `utils/task_metrics.py` 中 `build_terminal_summary` 为准更新本文档。
